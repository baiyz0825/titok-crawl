## 十、验证码输入框陷阱 - 双输入框问题 ⚠️⚠️⚠️（2026-03-13 重大发现）

### 10.1 问题描述

**现象**：后端使用 `page.query_selector('#button-input')` 无法找到正确的验证码输入框，导致验证码无法填入。

### 10.2 根本原因

通过 Chrome DevTools 实时观察发现，**页面上存在两个 `#button-input` 元素**：

| 元素 | ID | Class | Value | 状态 | 说明 |
|------|----|----|-------|------|------|
| 第一个 | `#button-input` | `tnpNAdqe _Yqor1vk` | `""` (空) | 不可见 | 旧输入框/隐藏 |
| **第二个** | `#button-input` | `input-lrnhMm` | 用户输入的值 | **可见、可编辑** | **真正的输入框** |

### 10.3 querySelector 的陷阱

```javascript
// ⚠️ 这只返回第一个元素！
document.querySelector('#button-input')  // 返回空的那个旧输入框

// ✅ 正确做法
document.querySelectorAll('#button-input')[1]  // 返回第二个（真正的）
```

**问题核心**：
- Playwright 的 `page.query_selector('#button-input')` 等同于 `document.querySelector()`
- 总是返回第一个匹配的元素
- 第一个元素是空的、不可用的
- 所以验证码永远填不进去！

### 10.7 推荐的完整实现

```python
async def fill_verify_code(self, page: Page, code: str) -> bool:
    """填入验证码并点击确认按钮

    ⚠️ 必须处理双输入框问题！
    """
    try:
        # 步骤1: 等待验证窗口
        logger.info("⏳ 等待验证窗口...")
        await page.wait_for_selector('#uc-second-verify', timeout=5000)

        # 步骤2: 使用多种策略查找正确的输入框
        logger.info("🔍 查找验证码输入框...")

        # 策略1: 通过 class 匹配（最可靠）
        inp = await page.query_selector('#button-input:not(.tnpNAdqe)')

        # 策略2: 如果失败，使用 querySelectorAll
        if not inp:
            all_inputs = await page.query_selector_all('#button-input')
            if len(all_inputs) >= 2:
                inp = all_inputs[1]

        # 策略3: 如果还失败，通过可见性查找
        if not inp:
            inp = await page.locator('input[placeholder="请输入验证码"]').first

        if not inp:
            logger.error("❌ 未找到可用的验证码输入框")

        logger.info(f"🔍 输入框当前值: '{filled_value}'")

        # 步骤5: 等待并点击验证按钮
        logger.info("⏳ 等待验证按钮可点击...")

        # 使用 JavaScript 点击（更可靠）
        result = await page.evaluate("""
            () => {
                const verifyWindow = document.getElementById('uc-second-verify');
                if (!verifyWindow) {
                    return { success: false, reason: 'window_not_found' };
                }

                const allDivs = Array.from(verifyWindow.querySelectorAll('div'));
                const verifyBtn = allDivs.find(el =>
                    el.textContent?.trim() === '验证' &&
                    el.classList.contains('primary-Npo6wt') &&
                    !el.classList.contains('disabled')
                );

                if (verifyBtn) {
                    verifyBtn.click();
                    return { success: true, reason: 'clicked' };
                }

                return { success: false, reason: 'button_disabled_or_not_found' };
            }
        """)

        if result.get('success'):
            logger.info("✅ 成功点击验证按钮")
            return True
        else:
            logger.warning(f"⚠️ 点击失败: {result.get('reason')}")
            return False

    except Exception as e:
        logger.error(f"❌ 填入验证码时出错: {e}")
        return False
```

## 十一、用户信息获取优化 - SSR数据直接读取（2026-03-13 优化）

### 11.1 问题背景

登录成功后需要采集当前用户信息，但传统的 API 拦截方式存在以下问题：

1. **当前用户的 API 端点不同**：
   - 其他用户：`user/profile/other` API
   - 当前用户：可能使用 `user/profile/self` 或 SSR 数据

2. **API 拦截失败率高**：
   - 需要重新导航到用户页面
   - 需要等待 API 响应
   - 网络延迟可能导致超时

3. **日志显示的问题**：
   ```
   First login for user 7616447372475139622, scraping profile...
   Empty user info for 7616447372475139622
   Failed to scrape profile for 7616447372475139622
   ```

### 11.2 SSR数据结构分析

通过 Chrome DevTools 分析发现，抖音页面使用 SSR (Server-Side Rendering)，所有用户数据都在 `window.SSR_RENDER_DATA` 中：

### 11.3 优化方案

#### 方案对比

| 方案 | 速度 | 可靠性 | 缺点 |
|------|------|--------|------|
| **SSR数据读取** | ⚡⚡⚡ 最快 | ✅ 高 | 需要已在用户页面 |
| **API拦截(self)** | ⚡⚡ 中 | ✅ 高 | 需要等待API响应 |
| **API拦截(other)** | ⚡⚡ 中 | ✅ 高 | 需要导航+等待 |
| **数据库查询** | ⚡⚡⚡ 最快 | ✅ 高 | 仅限已有数据 |

#### 推荐实现策略

```python
async def scrape_profile(self, sec_user_id: str) -> User | None:
    """多策略用户信息获取"""

    # 策略1: 优先从SSR数据读取（当前用户，最快）
    current_user_id = await engine.get_current_user_id()
    is_current_user = (current_user_id == sec_user_id)

    if is_current_user:
        user_from_ssr = await self._get_user_from_ssr(sec_user_id)
        if user_from_ssr:
            await crud.upsert_user(user_from_ssr)
            return user_from_ssr

    existing_user = await crud.get_user(sec_user_id)
    if existing_user:
        return existing_user

    # 策略3: 使用API拦截（兜底）
    return await self._scrape_profile_from_api(sec_user_id)
```

### 11.4 SSR数据读取实现

```python
async def _get_user_from_ssr(self, sec_user_id: str) -> User | None:
    """从SSR_RENDER_DATA读取用户信息"""
    try:
        page = await engine.get_page()

        user_data = await page.evaluate("""
            () => {
                if (!window.SSR_RENDER_DATA?.app?.odin) {
                    return null;
                }

                const odin = window.SSR_RENDER_DATA.app.odin;
                return {
                    sec_user_id: odin.user_unique_id || odin.sec_user_id,
                    nickname: odin.nickname || odin.user_unique_id,
                    avatar_url: odin.avatar_url ||
                               odin.avatar_thumb?.url_list?.[0],
                    douyin_id: odin.sec_uid,
                    signature: odin.signature,
                    following_count: odin.following_count,
                    follower_count: odin.follower_count,
                    favoriting_count: odin.favoriting_count,
                    aweme_count: odin.aweme_count || 0,
                    total_favorited: odin.total_favorited
                };
            }
        """)

        if not user_data or not user_data.get('sec_user_id'):
            return None

        user_data['backend_id'] = sec_user_id
        return User(**user_data)

    except Exception as e:
        logger.debug(f"Failed to get user from SSR: {e}")
        return None
```
